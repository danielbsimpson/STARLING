# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

Adds a YouTube feed panel to S.T.A.R.L.I.N.G. triggered exclusively by **"Open YouTube Feed"** or **"View YouTube Feed"**. The panel fetches the 15 most recent videos from each configured channel via YouTube's public Atom/RSS feed (no API key required for Phase 1), displays them as thumbnail tiles in a CSS grid, and lets the user filter by channel, toggle between Long Form and Shorts, and sort by date or view count. The local LLM synthesises the feed in the background and Starling speaks a channel-by-channel briefing once synthesis completes. Phase 2 adds a YouTube Data API v3 key for video duration fetching (enabling reliable Shorts vs Long Form classification) and YouTube handle-to-channel-ID resolution. Phase 3 adds Google OAuth2 so the user's actual YouTube subscription list is auto-discovered from their account.

## 1. Requirements & Constraints

- **REQ-001**: The voice trigger MUST match only the exact phrases "Open YouTube Feed" or "View YouTube Feed" (case-insensitive). Phrases such as "show me YouTube", "YouTube", or "play a video" must NOT activate this panel.
- **REQ-002**: Phase 1 uses only `feedparser` (already in `requirements.txt`) and `httpx` (already in `requirements.txt`) — no new dependencies. Channels are configured via `YOUTUBE_CHANNELS` env var as comma-separated YouTube channel IDs (format: `UCxxxxxxxxxxxxxxxxxxxxxxxxx`).
- **REQ-003**: Each channel's RSS feed URL is `https://www.youtube.com/feeds/videos.xml?channel_id={CHANNEL_ID}`. YouTube returns a maximum of 15 most recent uploads per channel feed; no limit parameter is supported.
- **REQ-004**: Videos are displayed as thumbnail tiles in a CSS grid. Each tile must show: thumbnail image (loaded from `https://i.ytimg.com/vi/{VIDEO_ID}/mqdefault.jpg`), video title, channel name, view count, and human-readable age. A "SHORTS" badge must appear on videos classified as Shorts (Phase 2 required for classification).
- **REQ-005**: The panel must include a type filter row: **All** | **Long Form** | **Shorts**. The Shorts and Long Form filter buttons must be present from Phase 1 but visually disabled (greyed out, `data-requires-api="true"`, tooltip "Requires YOUTUBE_API_KEY") until Phase 2 is configured. When Phase 2 is active (API key present) the buttons become functional.
- **REQ-006**: The panel must include a sort control row: **Date** (default) | **Views** | **Channel**. All three sort modes must work in Phase 1 (view count is available in RSS `media:statistics`).
- **REQ-007**: The panel must include a per-channel filter tab bar: "All Channels" + one button per channel (showing channel name). Clicking a tab filters the tile grid to that channel only.
- **REQ-008**: Post-synthesis, Starling must automatically speak a spoken briefing via `enqueueSpeak()` with a channel-by-channel summary of recent uploads. No markdown in briefing text.
- **REQ-009**: The user's YouTube subscription list must be auto-discovered in Phase 3 via Google OAuth2, replacing the manual `YOUTUBE_CHANNELS` env var. Until Phase 3 is configured, the env var list is used.
- **REQ-010**: A refresh button must clear the backend cache and re-fetch all channel feeds.
- **REQ-011**: Close triggers: "close YouTube", "close feed", and standard exit phrases used by existing panels.
- **REQ-012**: Cache TTL must be configurable via `YOUTUBE_CACHE_SECONDS` env var (default 1800 seconds / 30 minutes).
- **REQ-013**: Clicking a video tile must open the video URL (`https://www.youtube.com/watch?v={VIDEO_ID}`) in the existing browser panel via `openBrowserPanel(url)` if available, or fall back to `window.open(url, '_blank')`.

- **SEC-001**: The `YOUTUBE_API_KEY` env var (Phase 2) and `YOUTUBE_CLIENT_SECRET` (Phase 3) must never be logged, echoed in API responses, or committed to version control.
- **SEC-002**: Channel IDs from env vars must be validated against the pattern `^UC[A-Za-z0-9_\-]{22}$` before being interpolated into URLs. Invalid channel IDs must be logged and skipped, not raise unhandled exceptions.
- **SEC-003**: Video titles and descriptions returned from the YouTube API must be set via `element.textContent` (not `innerHTML`) in the frontend to prevent XSS.
- **SEC-004**: Google OAuth2 tokens (Phase 3) must be stored in `backend/memory/youtube_token.json` with file permissions restricted to the current user. Tokens must never be included in any API response body.

- **CON-001**: YouTube RSS feeds return exactly 15 most recent videos per channel. There is no `limit` or `page` parameter in the RSS URL. Pagination is not possible via RSS alone.
- **CON-002**: YouTube RSS does not include video duration. Shorts vs Long Form classification requires YouTube Data API v3 (`videos.list?part=contentDetails`) — Phase 2 dependency.
- **CON-003**: YouTube Data API v3 free quota is 10,000 units per day. A `videos.list?part=contentDetails` request costs 1 unit per call regardless of the number of video IDs (up to 50 IDs per call). With 20 channels × 15 videos = 300 videos = 6 batch API calls = 6 units. Well within quota.
- **CON-004**: YouTube subscription discovery (Phase 3) requires a Google Cloud project with OAuth2 credentials (`youtube.readonly` scope). The OAuth2 consent screen must be published or the user's Google account must be added as a test user if the app is in testing mode.
- **CON-005**: The panel must reuse existing CSS variables and class conventions from `frontend/style.css`. No external CSS or JS libraries may be added.
- **CON-006**: `feedparser` may parse YouTube's `yt:` and `media:` namespace attributes differently across versions. All attribute accesses must use `.get()` with safe defaults; attribute names are documented in TASK-004 with fallback strategies.

- **GUD-001**: Follow the exact fetch/cache/background-synthesis/polling pattern established in `backend/news.py` and `frontend/news-panel.js`.
- **GUD-002**: Synthesis prompt must produce plain-prose output only — no markdown characters (`*`, `#`, `-` as bullets) since the output is spoken aloud by TTS.
- **GUD-003**: All panel JS public functions must use `export function` (same convention as news-panel.js, weather-panel.js, etc.).
- **GUD-004**: Thumbnails must have an `onerror` handler that hides the `<img>` element and shows a fallback placeholder `<div>` with the channel initial on load failure.

- **PAT-001**: Background synthesis: `BackgroundTasks.add_task(_run_synthesis_bg, videos, cache_key)` fired after `GET /youtube` returns, identical to the news.py synthesis pattern.
- **PAT-002**: Synthesis polling: frontend polls `GET /youtube/synthesised` every 3 seconds up to 40 attempts (same constants as news-panel.js `SYNTH_POLL_INTERVAL_MS = 3000`, `SYNTH_POLL_MAX = 40`).
- **PAT-003**: LLM injection for spoken briefing: `sendToOllama(prompt, { extraContext: [{ role: 'system', content: data.llm_context }] })` — same as news briefing pattern in app.js.

## 3. Alternatives

- **ALT-001**: Use `yt-dlp` Python package to extract video metadata including duration without an API key. Rejected because `yt-dlp` is a large dependency (100+ MB), it's designed for downloading media not metadata-only queries, and using it at scale may violate YouTube's Terms of Service for automated data extraction.
- **ALT-002**: Scrape the YouTube channel page HTML to detect Shorts (Shorts have a `/shorts/` path in their rendered card links). Rejected because HTML scraping is brittle, breaks on layout changes, and violates YouTube's ToS for automated scraping.
- **ALT-003**: Use `youtube-transcript-api` to fetch video transcripts for richer LLM synthesis. Deferred — this could be a future enhancement once the base panel is working, as it adds significant per-video API calls.
- **ALT-004**: Render the YouTube panel as a single scrollable list (like the news panel) instead of a tile grid. Rejected per user requirement — thumbnails as tiles are explicitly requested and provide better visual scannability for video content.
- **ALT-005**: Display 3-column tile grid instead of 2-column. The panel width in the current layout (side panel that expands from the right column) makes 2 columns more readable without truncating titles. 3-column layout is documented as a CSS variable option (`--yt-grid-cols: 2`) so it can be overridden by the user.
- **ALT-006**: Use a looser trigger such as "show me YouTube" or "open YouTube". Rejected per user specification — the trigger must be specific to prevent accidental activation when "YouTube" appears in normal conversation.
- **ALT-007**: Combine YouTube and Reddit into a single "Social" panel with tabs. Deferred to a future `feature-social-hub` plan. Each panel is implemented standalone first to keep complexity manageable.

## 7. Risks & Assumptions

- **RISK-001**: YouTube RSS feeds return only the 15 most recent uploads per channel. For active channels that post multiple times per day, older videos will be evicted quickly. Mitigation: this is a known RSS limitation; the cache TTL (30 min default) means users see "up to 15 recent videos per channel within the last 30 minutes". Future enhancement: use `GET /youtube/v3/search?channelId=...&type=video&order=viewCount` for top videos rather than most recent.
- **RISK-002**: The `yt:` and `media:` namespace fields in feedparser may have different attribute names across feedparser versions (e.g., `entry.yt_videoid` vs `entry['yt_videoid']`). Mitigation: TASK-004 documents two extraction paths with fallbacks for all critical fields. Validate with `assert isinstance(feed.entries[0].get('yt_videoid', ''), str)` in a startup check.
- **RISK-003**: YouTube Data API v3 free quota is 10,000 units/day shared across all API calls in the Google Cloud project. If subscription list fetching is added (Phase 3), `subscriptions.list` costs 1 unit per call. With 30-minute caching, quota consumption is negligible for personal use. Monitor via Google Cloud Console.
- **RISK-004**: Google OAuth2 "Web application" credentials require the consent screen to be published (production) or the user's Google account added as a "Test user" (testing mode). Failure to do this correctly results in `access_denied` during the OAuth flow. Mitigation: document setup steps clearly in `plan/GMAIL.md` (which covers a parallel OAuth flow) and reference them.
- **RISK-005**: YouTube Shorts classification by duration (≤ 180 seconds) is a heuristic. Some regular videos under 3 minutes will be falsely classified as Shorts; some Shorts uploaded before the 3-minute limit extension will be correctly classified. A future improvement is to check the `shorts_url` via HTTP HEAD request (a 200 response means it's a Short; a redirect means it's not), but this adds 1 network call per video.
- **RISK-006**: The `thumbnail_url` from feedparser (`entry.media_thumbnail[0]['url']`) may be the lower-resolution `hqdefault.jpg`. For tile display, the plan substitutes `mqdefault.jpg` (320×180) by reconstructing the URL from `video_id` as `https://i.ytimg.com/vi/{video_id}/mqdefault.jpg`. If `video_id` extraction fails, the feedparser thumbnail URL is used as fallback.

- **ASSUMPTION-001**: The user's YouTube channel IDs are standard `UC...` format (24 characters). Old-style username-based channels that use `/user/` paths instead of `/channel/` paths may not have a `UCxxxxxx` channel ID in the usual sense; these require the Data API to resolve to a channel ID. Phase 2 handle resolution covers this.
- **ASSUMPTION-002**: The LLM model is capable of producing valid JSON when instructed. If it fails to produce valid JSON for the synthesis response, `_synthesise_videos()` returns `None` gracefully and the panel displays raw video tiles without a spoken briefing.
- **ASSUMPTION-003**: The browser-panel's `openBrowserPanel` function is exported from `browser-panel.js` and accepts a URL string as its argument (consistent with how other panels use it).
- **ASSUMPTION-004**: YouTube's public Atom/RSS feeds remain publicly accessible without authentication. This has been stable since at least 2015 and is not covered by the 2023 API access changes (which affected third-party apps using the Reddit/Twitter APIs). The RSS endpoint is separate from the Data API.
